## рҹӣҚпёҸ Overview

**Eissa** is a production-ready luxury fashion e-commerce application. It covers the complete shopping experience вҖ” from browsing paginated product collections and filtering by category, through a Stripe-hosted checkout, order creation, and post-purchase confirmation вҖ” and is backed by a full AWS 3-tier infrastructure provisioned entirely with Terraform, operated with Ansible, and deployed automatically via GitHub Actions CI/CD.

---

## вңЁ Features

### Storefront
- рҹҸ  **Home page** вҖ” animated hero, scrolling marquee, category grid, featured products, editorial split panels
- рҹ—ӮпёҸ **Product catalogue** вҖ” server-side paginated grid, URL-driven page state (`?page=N`)
- рҹ”Қ **Category filtering** вҖ” dedicated pages per category with independent pagination
- рҹ“„ **Product detail** вҖ” thumbnail gallery, accordion details section, recommended products
- рҹӣ’ **Cart drawer** вҖ” slide-in panel with quantity controls, free-shipping progress bar, real-time totals
- рҹҰҙ **Skeleton loaders** вҖ” 12 shimmer skeletons matching every loading state, zero layout shift

### Authentication
- рҹ”җ **JWT auth** вҖ” 15-minute access tokens + 7-day refresh tokens
- рҹҚӘ **httpOnly cookies** вҖ” XSS-safe, automatic silent refresh via Axios interceptor
- рҹ—„пёҸ **Redis session store** вҖ” refresh tokens stored server-side for instant revocation

### Checkout & Payments
- рҹ’і **Stripe hosted checkout** вҖ” PCI-compliant, card data never touches your server
- рҹҺҹпёҸ **Coupon codes** вҖ” percentage-discount coupons, per-user single-use enforcement
- рҹҺҒ **Auto-generated coupons** вҖ” orders over $200 generate a 10% gift coupon automatically
- вң… **Order confirmation** вҖ” verifies Stripe session, creates order, deactivates used coupon

### Performance
- вҡЎ **Redis caching** вҖ” featured products cached to avoid repeated DB queries
- рҹ“Ұ **Server-side pagination** вҖ” parallel `skip/limit` + count query
- рҹ–јпёҸ **Cloudinary** вҖ” image hosting and optimisation

### Infrastructure & DevOps
- рҹҸ—пёҸ **Terraform** вҖ” 11 modules covering the entire AWS stack (VPC, ECS, ALB, DocumentDB, ElastiCache, ECR, CloudFront, Route 53, ACM, WAF, IAM)
- рҹӨ– **Ansible** вҖ” 4 playbooks (deploy, rollback, seed-db, health-check), 2 roles (ecs-deploy, smoke-test)
- рҹҡҖ **GitHub Actions** вҖ” 3 workflows (frontend, backend, terraform) with OIDC вҖ” no stored AWS keys
- рҹҗі **Docker Compose** вҖ” single command spins up all 4 services locally
- рҹҸҘ **Health checks** вҖ” Docker + AWS ALB health checks on every service
- рҹ”’ **WAF + Shield** вҖ” OWASP managed ruleset, rate limiting, IP reputation blocking
- рҹ“Ҡ **CloudWatch** вҖ” Container Insights, alarms for 5xx rate, ECS CPU, DB connections, cost budget

## рҹ”җ Authentication Flow

```
Sign In вҶ’ generateToken(userId)
       вҶ’ accessToken (JWT, 15 min, httpOnly cookie)
       вҶ’ refreshToken (JWT, 7 days, httpOnly cookie)
       вҶ’ Redis SET refresh_token:{userId} EX 7days

Silent refresh (Axios interceptor):
  401 response вҶ’ POST /auth/refresh-token вҶ’ verify JWT + Redis match
              вҶ’ new accessToken cookie вҶ’ retry original request
```

## рҹҸ—пёҸ Infrastructure

![Infrastructure](./frontend/public/aws_infrastructure_diagram.svg)

### AWS Architecture

3-tier architecture in AWS us-east-1:

| Tier | Subnet type | Services |
|---|---|---|
| Presentation | Public | ALB, NAT Gateways, Internet Gateway |
| Application | Private | ECS Fargate (frontend + backend), Secrets Manager |
| Data | DB (isolated) | DocumentDB, ElastiCache Redis, S3 |

CloudFront + WAF + Route 53 sit outside the VPC as global edge services.

### Terraform Modules

| Module | Creates |
|---|---|
| `vpc` | VPC, 9 subnets (3Г—public/private/db), IGW, NAT gateways, route tables, security groups, VPC Flow Logs |
| `ecs` | ECS Fargate cluster, frontend + backend task definitions, services, auto-scaling (CPU 70%) |
| `alb` | Application Load Balancer, target groups, HTTPS listener (TLS 1.3), HTTPвҶ’HTTPS redirect |
| `documentdb` | MongoDB-compatible cluster (primary + replica), KMS encryption, automated backups |
| `elasticache` | Redis 7 cluster, TLS, auth token, secrets |
| `ecr` | Frontend + backend image repositories, lifecycle rules, scan on push |
| `cloudfront` | CDN distribution, SPA error routing (404вҶ’index.html), security response headers |
| `route53` | Hosted zone, ALIAS records (apex + www), health check |
| `acm` | Wildcard TLS certificate, auto-DNS validation, auto-renewal |
| `waf` | WebACL, rate limiting, OWASP common rules, known bad inputs, IP reputation |
| `iam` | ECS task execution role, ECS task role, GitHub Actions OIDC role (no static keys) |

### Deploying to AWS

```bash
# 1. Bootstrap (once only вҖ” creates S3 state bucket + DynamoDB lock table)
./bootstrap.sh

# 2. Fill in production tfvars
cp infrastructure/terraform.tfvars.example infrastructure/terraform.tfvars
# Edit: domain_name, alert_email

# 3. Initialise and apply
cd infrastructure
terraform init
terraform apply -var-file=environments/production/terraform.tfvars

# 4. Push initial Docker images to ECR
aws ecr get-login-password | docker login --username AWS --password-stdin \
  $(terraform output -raw frontend_ecr_url | cut -d'/' -f1)

# 5. Seed the database
cd ../ansible
ansible-playbook playbooks/seed-db.yml -i inventory/production.yml -e "seed_mode=--fresh"
```

See `RUNBOOK.md` for the complete phase-by-phase guide with every command.

## рҹ”„ CI/CD Pipeline

### Trigger rules

| Event | Workflows triggered |
|---|---|
| PR to `main` (frontend changes) | lint, TypeScript check, Vite build, Docker build (no push) |
| PR to `main` (backend changes) | lint, tests (real MongoDB + Redis containers) |
| PR to `main` (infra changes) | terraform fmt, validate, plan (posted as PR comment) |
| Merge to `main` (frontend) | lint вҶ’ build вҶ’ ECR push вҶ’ ECS rolling deploy вҶ’ smoke test |
| Merge to `main` (backend) | lint вҶ’ test вҶ’ ECR push вҶ’ ECS rolling deploy вҶ’ Ansible health check |
| Merge to `main` (infra) | terraform apply |

### Authentication

GitHub Actions uses **OIDC** to assume an AWS IAM role. No `AWS_ACCESS_KEY_ID` or `AWS_SECRET_ACCESS_KEY` is ever stored in GitHub secrets вҖ” GitHub exchanges a short-lived JWT for a temporary AWS session.

### Required GitHub secrets

```
AWS_ACCOUNT_ID
AWS_REGION
AWS_GITHUB_ACTIONS_ROLE_ARN   вҶҗ from: terraform output github_actions_role_arn
VITE_STRIPE_PUBLISHABLE_KEY
```
